# Test Strategy

## 1. Purpose

This document describes the test strategy for the Regulated QA Automation Demo project.

The purpose of this project is to demonstrate practical UI and API automation using Playwright and TypeScript, with emphasis on maintainable test structure, repeatable execution, and objective evidence generation.

## 2. Scope

This strategy covers the current demo automation suite, including:

* UI login flow automation
* UI negative login scenario automation
* API response validation
* API positive and negative scenarios
* HTML report generation
* Local Playwright execution

## 3. Out of Scope

The following areas are not covered in the current version of the demo project:

* Full production-grade validation
* Load or performance testing
* Security or penetration testing
* Accessibility testing
* Full cross-browser compatibility matrix
* Mobile browser testing
* Test data lifecycle management
* Real user data or production systems
* Formal regulatory submission documentation

## 4. Test Approach

The automation suite is divided into two main layers:

| Layer     | Purpose                                                                   |
| --------- | ------------------------------------------------------------------------- |
| UI tests  | Validate critical browser-based user workflows                            |
| API tests | Validate direct service/API behavior, status codes, and response payloads |

The UI tests use a Page Object Model to keep selectors and page actions separate from test logic.

The API tests use Playwright request testing to validate backend/API behavior without opening a browser.

## 5. Risk-Based Rationale

The current test cases were selected based on simple risk-based prioritization.

Higher-priority coverage was given to flows where failure would affect user access, expected system behavior, or confidence in basic regression stability.

| Area                      | Risk Rationale                                                      | Covered By |
| ------------------------- | ------------------------------------------------------------------- | ---------- |
| Valid login               | Failure blocks user access to the application                       | TC-UI-001  |
| Locked-out login          | Failure may allow incorrect access or provide unclear user feedback | TC-UI-002  |
| User API response         | Failure may indicate broken API contract or missing expected fields | TC-API-001 |
| Create post API flow      | Failure may indicate inability to accept valid payloads             | TC-API-002 |
| Invalid endpoint handling | Failure may indicate incorrect error handling                       | TC-API-003 |

## 6. Automation Selection Criteria

Test cases were selected for automation when they were:

* Repeatable
* Suitable for unattended execution
* Linked to clear expected results
* Useful for regression detection
* Reasonably stable against the selected public demo systems
* Easy to understand from test name and test ID

## 7. Evidence

The main evidence generated by the automation suite is the Playwright HTML report.

Additional evidence may include screenshots, videos, and traces when failures occur, based on the Playwright configuration.

## 8. Known Limitations

This project uses public demo systems. Failures may occasionally occur due to network availability, external service changes, or demo-site behavior outside the control of this repository.

The current test suite is intentionally limited and is not intended to represent full product validation.

## 9. Conclusion

The current automation strategy demonstrates a practical foundation for UI and API regression testing, with clear test IDs, maintainable structure, and evidence-oriented execution.

Future improvements may include additional workflow coverage, improved API schema validation, CI execution evidence, and formal traceability documentation.
